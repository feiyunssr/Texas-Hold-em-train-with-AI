# 工程方案评审（2026-04-23）

**状态：DONE_WITH_CONCERNS**

## 评审对象

- [产品需求文档](./PRD.md)
- [Office Hours 决策记录（2026-04-22）](./office-hours-2026-04-22.md)

## 结论

产品方向是对的，`训练牌桌 + 行动前 AI 建议` 也确实应该作为第一主航道；但在真正开工前，当前方案还缺少几项必须先锁死的工程决策。最大风险不是“做不出功能”，而是：

- 实时建议链路阻塞主流程，导致训练体验卡顿
- AI 对手与 AI 教练共享上下文时发生信息泄漏
- 手牌历史、建议快照、标签和计费无法可靠对账

如果先补齐这些边界，再进入开发，v1 的工程落地是可控的。

## 主要问题

### P0. 缺少实时建议的延迟预算、超时策略和降级路径

相关来源：

- PRD 第 29-33 行
- PRD 第 107-128 行
- Office Hours 第 68 行

现在只定义了“用户可以在决策点请求 AI 建议”，但没有定义：

- 可接受的响应时间上限
- 超时后是否收费
- 请求过程中牌局是否冻结
- 用户重复点击如何处理

如果这一点不先定，后端、前端、计费和埋点都会各做各的，最后一定返工。

**建议直接锁定：**

- v1 `AI 建议` 目标延迟：`p95 <= 1.5s`
- 硬超时：`2.5s`
- 超时或失败：不扣点，前端展示“本次建议未完成，可继续行动或稍后重试”
- 请求期间当前决策点冻结，禁止状态继续推进
- 同一决策点只允许 `1` 次正式建议请求，额外追问留到后续版本

### P0. “共用同一套结构化牌局上下文模型”表述不够精确，存在信息泄漏风险

相关来源：

- PRD 第 54-55 行
- PRD 第 105 行
- PRD 第 111-128 行
- Office Hours 第 77 行

现在文档强调 AI 对手和 AI 教练共用上下文模型，但没有明确“共用的是同一个事实源”，还是“共用同一份序列化 payload”。这两者差别很大。

如果直接把同一份对象喂给不同 AI，很容易出现：

- 对手拿到不该看到的隐藏手牌
- 教练拿到带内部状态标记的系统字段
- 后续重放时无法证明当时 AI 到底看到了什么

**建议直接锁定：**

- 系统内部只有一份 `canonical hand state`
- 从这份事实源派生三种只读视图：
  - `public table view`
  - `seat-private bot view`
  - `hero-coach view`
- 所有 AI 请求都必须保存“请求快照”而不是只保存最终答案
- 任何 AI 都不能直接读取数据库里的完整手牌对象

### P0. 训练资产模型缺少“不可变事件流 + 版本化快照”定义

相关来源：

- PRD 第 57 行
- PRD 第 128 行
- PRD 第 173-188 行
- PRD 第 229-245 行
- Office Hours 第 63-64 行

当前文档已经知道要保存很多数据，但还没有定义保存方式。对这个产品来说，普通 CRUD 表不够用，因为后续要支撑：

- 回放
- 复盘
- 标签统计
- 训练诊断
- 计费对账

**建议直接锁定：**

- 用 append-only 的 `hand_event_log` 作为权威时间线
- 每个用户决策点生成 `decision_snapshot`
- 每次 AI 输出单独保存为 `ai_artifact`
- 标签使用 `label_assignment`，并记录标签版本
- 所有结构化对象都带 `schema_version`

这样后续即使提示词、标签体系或教练输出结构升级，也不会污染旧数据。

### P1. 点数计费的扣费时机与幂等性未定义

相关来源：

- PRD 第 56 行
- PRD 第 109 行
- PRD 第 132 行
- PRD 第 247-258 行
- Office Hours 第 72 行

收费能力很多，但没有明确计费事件的原子性边界。这里如果处理得随意，最先坏的不是商业化，而是用户信任。

**建议直接锁定：**

- `AI 建议` 按“成功返回并持久化结果”扣费
- 所有收费请求都带 `request_id`
- 同一个 `request_id` 只能结算一次
- 超时、模型错误、存储失败均不扣费
- 审计日志必须能串起：`request_id -> AI 请求 -> AI 响应 -> 账务流水`

### P1. 第一里程碑仍然偏宽，需要工程上进一步收口

相关来源：

- Office Hours 第 57-64 行
- PRD 第 130-156 行
- PRD 第 218-245 行

当前文档已经说明“完整骨架，主航道优先”，但工程上仍然容易不自觉把复盘、长期统计、场景训练一起做深。

**建议直接锁定：**

- 第一里程碑只做“能完整打完一手牌 + 用户可请求一次 AI 建议 + 建议快照可保存 + 手牌结束后可触发基础复盘”
- 长期统计在 v1 只保留数据入库与最基础筛选
- 场景训练不进入第一里程碑的关键路径

## 推荐架构

推荐采用 **单体仓库 + 清晰领域分层**，不要一开始拆微服务。

### 1. 领域核心：规则引擎

职责：

- 桌面配置
- 发牌
- 行动轮转
- 合法动作生成
- 筹码、底池、边池结算
- 手牌结束判定

要求：

- 纯函数或接近纯函数
- 输入同样事件序列，输出必须完全一致
- 不依赖模型、不依赖数据库、不依赖 UI

这是整个产品最该“先做对”的部分。

### 2. 会话层：训练牌桌运行时

职责：

- 创建训练桌
- 维护当前手牌会话
- 处理真人动作
- 驱动 AI 对手行动
- 在用户决策点挂起并等待用户操作或 AI 建议请求

建议使用：

- API 做控制面
- WebSocket 做桌面实时同步

### 3. AI 编排层

职责：

- 接收来自牌桌运行时的标准化请求
- 组装不同 AI 的上下文视图
- 管理超时、重试、熔断和结果落库
- 统一返回结构化输出

这里不要把“提示词拼接”散落在业务代码里，应单独做成可版本化模块。

### 4. 训练资产层

职责：

- 保存手牌事件流
- 保存决策快照
- 保存 AI 建议和复盘产物
- 保存问题类型、用户标签和标签版本

这里的目标不是“先能存”，而是“后面还能可靠回放与统计”。

### 5. 读模型层

职责：

- 历史手牌列表
- 单手回放
- 训练问题聚合
- 基础筛选与检索

读模型可以从事件流异步投影，不要反过来让统计查询侵入规则引擎。

## 推荐的数据边界

最低限度建议拆成以下核心对象：

- `table_config`
- `table_seat_profile`
- `hand`
- `hand_event_log`
- `decision_snapshot`
- `ai_artifact`
- `label_definition`
- `label_assignment`
- `wallet_ledger`

其中：

- `hand_event_log` 是事实
- `decision_snapshot` 是“某个决策点当时看到了什么”
- `ai_artifact` 是“AI 当时说了什么”
- `wallet_ledger` 是“这次收费怎么结算的”

不要把这些信息揉进一张“万能手牌表”。

## 推荐默认决策

为避免继续悬空，建议直接采用以下默认值：

1. `AI 建议` 展示形式：主推荐 + 次推荐，不做更复杂的多分支树。
2. 同一决策点连续请求：v1 不开放多轮追问，只保留一次正式建议。
3. 与建议一致/不一致反馈：v1 可在手牌结束后展示，不要在实时决策时增加认知负担。
4. 计费方式：按次成功建议收费，后续再扩订阅包。
5. 首版对手 AI：先做固定人格原型，不做在线自适应。

## 里程碑建议

### M0. 规则引擎落地

交付标准：

- 纯规则引擎可独立跑完整手 NLHE 现金局
- 支持 `4-12` 人、盲注、Ante、Straddle
- 合法动作与结算可测试

### M1. 单桌训练闭环

交付标准：

- 单桌 Web 对局跑通
- AI 对手可以基于合法动作持续行动
- 用户在自己的决策点可以行动
- 完整手牌事件被持久化

### M2. 行动前 AI 建议闭环

交付标准：

- 用户请求 AI 建议
- 请求期间决策点冻结
- 结构化建议结果返回前端
- 建议快照、AI 输出、计费流水全部可追踪

### M3. 局后复盘与基础历史页

交付标准：

- 已结束手牌可触发复盘
- 单手回放可查看
- 历史列表可按基础条件筛选

场景训练、长期统计洞察、深度分析放到后续里程碑。

## 测试门槛

在这个项目里，测试不是锦上添花，而是产品正确性的主体。

必须至少覆盖：

- 规则引擎单元测试
- 合法动作快照测试
- 结算与边池测试
- 事件流重放测试
- AI 视图脱敏测试
- 计费幂等测试
- WebSocket 会话断线恢复测试

如果这些没建立，后面每加一个 AI 功能都会让系统更难验证。

## 建议的技术策略

如果现在从零启动，我建议：

- 使用 TypeScript 统一前后端和领域模型
- 采用 monorepo，但保持单部署单元优先
- 把规则引擎做成独立 package
- 数据库优先选择 PostgreSQL
- 实时通信使用 WebSocket
- AI 接入统一走一个网关模块，不让业务层直接调用模型 SDK

这里最重要的不是“技术栈时髦”，而是：

- 领域模型只有一份
- 状态推进可重放
- AI 边界可审计

## 最终判断

这个产品现在不是缺想法，而是缺工程约束。只要先锁定以下四件事，就可以进入实施：

1. 实时建议链路的延迟和失败语义
2. canonical state 与多视图脱敏模型
3. append-only 手牌事件流与 AI 快照存储
4. 点数计费的成功边界与幂等规则

这四项如果不先定，后面代码写得越多，返工越重。
